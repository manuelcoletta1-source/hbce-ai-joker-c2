## 1. Purpose

This document defines the project-domain governance map of AI JOKER-C2.

AI JOKER-C2 is the identity-bound cognitive command runtime of the HERMETICUM B.C.E. ecosystem.

It connects three primary domains:

1. MATRIX
2. CORPUS ESOTEROLOGIA ERMETICA
3. APOKALYPSIS

The purpose of this map is to define how the runtime should classify requests, documents, code operations, editorial operations, strategic operations and governance operations according to their active project domain.

Project-domain classification is required because AI JOKER-C2 must not treat every request as generic text generation.

Every relevant operation should be routed through the correct domain before policy, risk, decision, EVT, ledger and verification logic are applied.

9. Classification Priority

When more than one domain appears, classify using the following priority rules.

Rule 1: Explicit User Naming

If the user explicitly names a domain, use that domain.

Example:

"Update MATRIX B2B positioning"

Classification:

MATRIX

Rule 2: Explicit Multi-Domain Naming

If the user names two or more domains, use MULTI_DOMAIN.

Example:

"Connect MATRIX, CORPUS and APOKALYPSIS inside AI JOKER-C2"

Classification:

MULTI_DOMAIN

Rule 3: GitHub System Files

If the request concerns repository-level system files such as README, ARCHITECTURE, GOVERNANCE, EVT_PROTOCOL, PROTOCOL, SECURITY, COMPLIANCE, ROADMAP or system manifest, use MULTI_DOMAIN when the file governs the whole ecosystem.

Rule 4: Technical Runtime

If the request concerns runtime implementation, classify according to the module purpose.

Examples:

File	Likely Domain

lib/project-domain-classifier.ts	MULTI_DOMAIN
lib/evt.ts	MULTI_DOMAIN
lib/policy-engine.ts	MULTI_DOMAIN
docs/B2B_OVERVIEW.md	MATRIX
docs/CORPUS_OVERVIEW.md	CORPUS_ESOTEROLOGIA_ERMETICA
docs/APOKALYPSIS_OVERVIEW.md	APOKALYPSIS


Rule 5: Ambiguous Safe Requests

If the request is safe but domain is unclear, use GENERAL or infer the nearest domain from context.

Rule 6: Ambiguous Sensitive Requests

If the request is sensitive and the project domain is unclear, use ESCALATE or DEGRADE.

Do not treat ambiguity in sensitive contexts as permission.

14. Project-Domain Classifier Logic

A simplified classifier may follow this logic:

1. Normalize input.
2. Detect explicit domain names.
3. Detect domain-specific keywords.
4. Detect repository file context.
5. Detect technical module context.
6. Detect strategic, editorial or governance intent.
7. Assign MATRIX, CORPUS_ESOTEROLOGIA_ERMETICA, APOKALYPSIS, GENERAL or MULTI_DOMAIN.
8. Attach confidence.
9. Attach reasons.
10. Pass result to policy and risk layers.

{
  "project_domain": "MULTI_DOMAIN",
  "active_domains": [
    "MATRIX",

18. Domain Conflict Resolution

When domain classification is uncertain, use the following rules.

18.1 Safe Documentation

If the request concerns safe documentation and more than one domain may apply, classify as MULTI_DOMAIN.

18.2 Editorial Corpus Work

If the request concerns theoretical writing, glossary, volume structure or canonical formulas, classify as CORPUS_ESOTEROLOGIA_ERMETICA, unless the user explicitly connects it to APOKALYPSIS or MATRIX.

18.3 Historical Threshold Work

If the request concerns decay, exposure, cultural-political-social system analysis or Paradogma Alieno in the context of historical development, classify as APOKALYPSIS.

18.4 Institutional Infrastructure Work

If the request concerns Europe, infrastructure, AI governance, B2B, B2G, public administration, cybersecurity, cloud, data or energy, classify as MATRIX.

18.5 Runtime Governance Work

If the request concerns AI JOKER-C2 runtime, governance, EVT, policy, risk, protocol, security, compliance or roadmap, classify as MULTI_DOMAIN.

18.6 Sensitive Ambiguity

If the request is sensitive and classification is uncertain, use ESCALATE or DEGRADE.

23. Invariants

The following invariants must remain stable:

1. AI JOKER-C2 must not treat every request as generic.


2. MATRIX, CORPUS and APOKALYPSIS must remain distinct.


3. MULTI_DOMAIN must be used when the whole ecosystem is involved.


4. Project-domain classification must precede sensitive governance decisions.


5. EVT records must preserve project-domain metadata.


6. Repository-level governance files are usually MULTI_DOMAIN.


7. B2B and B2G documents usually belong to MATRIX.


8. Corpus theoretical work belongs to CORPUS_ESOTEROLOGIA_ERMETICA.


9. Historical threshold work belongs to APOKALYPSIS.


10. Sensitive ambiguity must not authorize execution.


11. Prohibited requests must be blocked.


12. Public communication should remain readable.


13. Internal conceptual coherence must not become external certification.


14. The model layer must not replace governance.


15. Human accountability must be preserved.
