## 7. Data Classes

AI JOKER-C2 should classify data before processing.

Suggested data classes:

```txt
PUBLIC = ordinary public or non-sensitive content
INTERNAL = controlled internal project or runtime context
CONFIDENTIAL = restricted business, institutional or legal material
SENSITIVE = protected information requiring minimization
SECRET = credentials, keys, tokens or private security material
PERSONAL = personal data or identifiers
CIVIC_SENSITIVE = civic participation, eligibility or public decision data
DEMOCRATIC_CHOICE = vote content or civic-choice content
SECURITY_SENSITIVE = security findings, logs, vulnerabilities or infrastructure details
CRITICAL_OPERATIONAL = live systems, critical infrastructure or operational controls
UNSUPPORTED = executable, binary or unsafe file types
UNKNOWN = insufficient information to classify safely
```

The runtime must not treat unknown data as safe by default in sensitive contexts.

---

## 8. Data Handling Rules

AI JOKER-C2 should follow these handling rules.

```txt
PUBLIC = process normally
INTERNAL = process with controlled handling
CONFIDENTIAL = minimize and review before external use
SENSITIVE = minimize, redact or reference
SECRET = do not expose, do not repeat, recommend rotation where leaked
PERSONAL = minimize, avoid unnecessary reproduction, preserve purpose limitation
CIVIC_SENSITIVE = minimize and audit
DEMOCRATIC_CHOICE = do not link to identity; escalate or block depending on context
SECURITY_SENSITIVE = defensive-only handling and review
CRITICAL_OPERATIONAL = escalate before operational reliance
UNSUPPORTED = reject or treat as inert metadata only
UNKNOWN = treat conservatively
```

The general rule is:

```txt
Process only what is necessary.
Expose only what is safe.
Store only what is justified.
Trace only what is needed for accountability.
```

## 9. Minimization Principle

Data minimization is mandatory.

The runtime should avoid:

```txt
storing full user documents when a reference is enough;
repeating credentials or secrets;
copying unnecessary personal data;
storing democratic choice content;
logging private payloads;
embedding sensitive content into public EVT records;
embedding sensitive content into public OPC proof views;
over-retaining file context;
using memory as uncontrolled storage.
```

The preferred pattern is:

```txt
reference;
summary;
hash;
metadata;
public-safe excerpt;
redacted view;
controlled internal audit view.
```

Minimization must not remove accountability.

The goal is to preserve evidence without unnecessary exposure.

## 10. Hash and Reference Strategy

For sensitive or large payloads, the runtime should use references and hashes.

Recommended approach:

```txt
large file = file metadata + hash + controlled reference
sensitive text = redacted summary + hash
private data = minimized reference + access-controlled audit note
public output = public-safe metadata
internal record = richer metadata only when justified
```

Hashing supports integrity, but it does not create legal certification.

Correct relation:

```txt
Hash supports verification.
EVT supports traceability.
OPC supports proof receipt.
Audit supports review.
Certification remains external.
```

## 21. Retention Strategy

The runtime should not retain data indefinitely without purpose.

Suggested retention principles:

```txt
short-lived session context for runtime interaction;
EVT metadata retained for traceability;
hashes retained for verification;
OPC proof receipts retained for audit;
sensitive payloads minimized or excluded;
secrets never retained intentionally;
democratic choice content never retained with identity;
file content retained only when necessary and authorized.
```

Retention must remain proportional to purpose.
