## 1. Purpose

This document defines the runtime model of AI JOKER-C2.

AI JOKER-C2 is the identity-bound cognitive command runtime of the HERMETICUM B.C.E. ecosystem.

It connects three primary domains:

1. MATRIX
2. CORPUS ESOTEROLOGIA ERMETICA
3. APOKALYPSIS

The runtime model defines how AI JOKER-C2 receives input, classifies the active domain, applies policy, assigns risk, determines human oversight, produces a decision, generates output, creates EVT records, preserves ledger continuity and supports verification.

AI JOKER-C2 is not a generic chatbot.

It is a governed runtime that transforms AI interaction into project-aware, policy-aware, risk-classified and event-traceable operational sequence.

5. Runtime States

AI JOKER-C2 should support the following runtime states.

State	Meaning

OPERATIONAL	Runtime is functioning normally
DEGRADED	Runtime is partially limited
BLOCKED	Runtime cannot proceed
INVALID	Runtime state is invalid or unverifiable

6. Runtime Pipeline

The runtime pipeline is divided into twelve stages.

Stage	Function

1. Identity	Bind operation to entity, IPR and runtime
2. Input	Receive and normalize request
3. Intent	Classify user purpose
4. Context	Classify operational context
5. Project Domain	Route to MATRIX, CORPUS, APOKALYPSIS, GENERAL or MULTI_DOMAIN
6. Policy	Apply allowed, restricted and prohibited boundaries
7. Risk	Assign operational risk class
8. Human Oversight	Determine review requirement
9. Decision	Produce ALLOW, BLOCK, ESCALATE, DEGRADE, AUDIT or NOOP
10. Execution	Generate only the permitted output
11. EVT	Record significant operation
12. Verification	Support reconstruction and continuity


The pipeline should be preserved across all future implementations.

7. Input Layer

user message;

uploaded text;

file metadata;

active document context;

repository operation request;

runtime diagnostic request;

governance request;

security request;

compliance request;

editorial request;

strategic request.


Input handling rules:

1. Normalize safely.


2. Do not execute uploaded content.


3. Do not trust filenames blindly.


4. Avoid exposing secrets.


5. Avoid logging sensitive payloads.


6. Preserve user intent.


7. Preserve file visibility limits.


8. Preserve enough metadata for classification.


9. Apply conservative handling to unknown data.


10. Trigger fail-closed for unsafe or ambiguous sensitive input.



Input does not authorize execution.

It only prepares the request for classification.

15. Policy Layer

The policy layer determines whether the requested operation is within scope.

Policy checks should consider:

project domain;

intent;

context;

data class;

user authority;

possible misuse;

dual-use implications;

human oversight requirement;

legal or compliance sensitivity;

security boundary;

EVT requirement.


Suggested policy outcomes:

Policy Outcome	Meaning

PERMIT	Operation is within scope
RESTRICT	Only limited safe support is allowed
REQUIRE_AUDIT	Operation should be recorded or reviewed
REQUIRE_REVIEW	Human review is required before continuation
PROHIBIT	Operation is outside scope or unsafe


Policy must precede sensitive execution.

16. Risk Layer

The risk layer assigns operational sensitivity.

Risk Class	Meaning	Default Handling

LOW	Ordinary safe request	ALLOW
MEDIUM	Operationally relevant request	ALLOW or AUDIT
HIGH	Sensitive or strategic request	ESCALATE or DEGRADE
CRITICAL	Strict review required	ESCALATE or BLOCK
PROHIBITED	Unsafe or outside scope	BLOCK
UNKNOWN	Cannot be safely classified	ESCALATE or BLOCK


Risk is determined by both explicit request and likely operational effect.

Unknown risk in sensitive contexts must trigger fail-closed behavior.


---

17. Human Oversight Layer

Human oversight determines whether a responsible human must review the operation.

Oversight states:

State	Meaning

NOT_REQUIRED	Ordinary safe operation
RECOMMENDED	Human review useful but not mandatory
REQUIRED	Human review required before continuation
COMPLETED	Human review completed
REJECTED	Human review rejected the operation
ESCALATED	Further authority or specialized review required


High-impact contexts require oversight.

Required oversight cannot be bypassed by model output.


---

The decision layer converts policy, risk and oversight into a runtime action.

Allowed decisions:

Decision	Meaning

ALLOW	Operation may proceed
BLOCK	Operation is prohibited or unsafe
ESCALATE	Human review or authority required
DEGRADE	Limited safe support only
AUDIT	Operation should be recorded or reviewed
NOOP	No operational action taken

The decision must be produced before or during execution.

It must not be invented after execution.

19. Execution Layer

Execution may include:

answering a question;

producing a GitHub-ready file;

creating governance documentation;

producing a risk register;

creating an incident report;

Execution must follow the decision.

Decision	Execution Rule

ESCALATE	Provide safe preparatory support only
DEGRADE	Provide limited safe output

NOOP	Take no operational action

21. Ledger Layer

The ledger preserves EVT continuity.

Ledger principles:

1. Events are append-only.


2. Historical events are not silently rewritten.


3. Corrections are new events.


4. Previous-event references are preserved.


5. Hashes are reproducible where possible.


6. Project-domain metadata is preserved.


7. Public and internal views are separated.


8. Sensitive payloads are minimized.


9. Audit status is explicit.


10. Verification status is explicit.



Ledger rule:

Corrections are new events, not silent mutations.


---

22. Verification Layer

Verification allows reconstruction.

A verifier should check:

event existence;

required fields;

previous event reference;

timestamp;

identity;

runtime state;

project domain;

context class;

risk class;

decision;

operation status;

hash consistency;

correction or supersession state;

public-safe disclosure.


Verification status values:

Status	Meaning

VERIFIABLE	Sufficient data for verification
PARTIAL	Useful but incomplete
INVALID	Failed structural validation
UNVERIFIED	Not yet checked
ANCHORED	Externally anchored
SUPERSEDED	Corrected by later event


Verification is reconstruction.

It is not legal certification.


---

23. Fail-Closed Layer

Fail-closed is the safety boundary.

Fail-closed activates when:

identity is missing;

authority is unclear;

project domain cannot be classified in a sensitive context;

policy cannot be applied;

risk cannot be classified;

human review is required but absent;

request is prohibited;

data is secret or unsafe to process;

output may enable abuse;

operation may affect real systems without authorization;

EVT cannot be generated for a significant operation;

runtime state is invalid.


Fail-closed rule:

Uncertainty in sensitive operations does not authorize execution.

Fail-closed outcomes:

BLOCK;

ESCALATE;

DEGRADE;

AUDIT;

NOOP.
