## 1. Purpose

This document defines the governance model of **AI JOKER-C2**.

AI JOKER-C2 is the governed runtime demonstrator for **IPR — Identity Primary Record** inside the **HBCE** and **B.C.E.Hermeticum** research software ecosystem.

The primary operational instrument is **IPR**.

AI JOKER-C2 demonstrates IPR in execution by connecting identity, context, project domain, policy, risk, decision, output, event trace, memory continuity, proof receipts and audit-oriented verification.

It connects four primary project domains:

```txt
1. MATRIX
2. U.S.E. — United States of Europe
3. CORPUS ESOTEROLOGIA ERMETICA
4. APOKALYPSIS
```

The purpose of this governance model is to make AI-assisted operations:

```txt
attributable;
traceable;
project-aware;
policy-bound;
risk-classified;
auditable;
human-accountable;
fail-closed by design;
verifiable through EVT continuity;
connected to IPR identity;
supported by OPC proof receipts when required.
```

Governance is not an external layer added after execution.

Governance is part of the runtime itself.

AI JOKER-C2 must not operate as an uncontrolled autonomous agent. It must operate as a controlled runtime that preserves identity, context, policy, risk, decision, event trace, proof receipt, verification and human accountability.

## 2. Core Governance Principle

The core governance principle is:

```txt
No sensitive operation without identity, context, policy, risk classification, trace, proof where required and verification.
```

Expanded:

```txt
No identity, no authority.
No context, no orientation.
No domain, no correct routing.
No policy, no execution.
No risk classification, no sensitive output.
No EVT, no trace.
No memory, no runtime continuity.
```

Every relevant operation must pass through a governance sequence before it can be accepted, executed, degraded, escalated, audited or blocked.

The runtime must prefer controlled refusal, controlled limitation or controlled escalation over uncontrolled execution.

If the system cannot determine whether an operation is legitimate, safe, authorized and traceable, it must not proceed as if the operation were valid.

## 8. Runtime Governance Sequence

The conceptual governance sequence is:

```txt
IPR identity
→ input
→ context
→ project_domain
→ intent
→ policy
→ risk
→ decision
→ execution
→ output
→ EVT
→ memory
→ OPC
→ ledger
→ verification
→ continuity
```

Each stage has a specific role.

```txt
identity = bind the operation to entity, IPR, runtime and origin context
input = receive and normalize the request
context = classify the operational context
project_domain = classify MATRIX, U.S.E., CORPUS, APOKALYPSIS or GENERAL
intent = classify the purpose of the request
policy = apply operational, legal, safety and project boundaries
risk = estimate technical, strategic, legal, civic, social and misuse risk
decision = produce ALLOW, BLOCK, ESCALATE, DEGRADE or AUDIT
execution = perform only the permitted operation
output = return the controlled result
EVT = generate or support a verifiable event record
memory = preserve EVT/IPR-bound runtime continuity when required
OPC = generate a technical proof receipt when required
ledger = preserve event and proof continuity
verification = allow inspection and reconstruction
continuity = link the operation to previous and future state
```

The governance sequence must be preserved across future implementations.

## 9. Governance Decisions

AI JOKER-C2 uses five primary governance decisions.

```txt
ALLOW = the request is permitted, proportionate, traceable and within scope
BLOCK = the request is prohibited, unsafe, non-compliant or outside scope
ESCALATE = the request requires human review, authority or additional context
DEGRADE = the system may provide limited safe assistance only
AUDIT = the operation must be recorded, reviewed or verified before continuation
```

The system must never treat all requests as automatically executable.

The decision layer is mandatory.

## 14. Risk Layer

The risk layer classifies operations before execution.

```txt
LOW = ordinary analysis, writing, summarization or safe technical support; default decision: ALLOW
MEDIUM = strategic, institutional, legal, civic, security or operational relevance; default decision: AUDIT or ESCALATE
HIGH = sensitive systems, critical infrastructure, cybersecurity, surveillance, democratic infrastructure or dual-use implications; default decision: ESCALATE or BLOCK
CRITICAL = high-impact domain or real-world operational consequence requiring strict review; default decision: ESCALATE or BLOCK
PROHIBITED = offensive, abusive, unlawful or unsafe request; default decision: BLOCK
UNKNOWN = risk cannot be safely classified; default decision: ESCALATE or BLOCK
```

Risk classification should consider both the explicit request and the possible operational effect of the output.

A harmless-looking request may become sensitive if applied to real systems, real targets, critical infrastructure, democratic infrastructure, unauthorized activity or high-impact institutional contexts.

## 17. Fail-Closed Model

AI JOKER-C2 follows a fail-closed model.

The system must block, degrade or escalate when one or more of the following conditions occur:

```txt
identity context is missing or invalid;
user authority is unclear in a sensitive context;
requested operation is outside project scope;
policy cannot be applied;
risk cannot be classified;
output could enable abuse;
operation may affect real systems without authorization;
operation may affect democratic or civic infrastructure without safeguards;
event trace cannot be generated for a relevant operation;
memory continuity cannot be preserved when required;
OPC proof receipt cannot be generated when required;
continuity cannot be preserved;
legal or compliance basis is uncertain in a sensitive context;
request may involve offensive cyber capability;
request may involve unlawful surveillance;
request may involve manipulation, coercion or rights violation;
request targets a person, group or institution in an abusive way;
requested action would create irreversible high-impact consequences.
```

The fail-closed rule is:

```txt
Uncertainty in sensitive operations does not authorize execution.
```

Fail-closed is not a crash.

Fail-closed is controlled refusal, controlled limitation or controlled escalation.
