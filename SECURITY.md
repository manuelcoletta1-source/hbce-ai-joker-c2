## 10. Vulnerability Reporting

Security reports should be handled responsibly.

Do not publish exploit details in public issues if the report could enable abuse.

Recommended reporting path:

```txt
1. Use GitHub private vulnerability reporting if available.
2. Use a private communication channel controlled by the repository owner.
3. Provide a concise technical description.
4. Include affected file, route, module or configuration.
5. Include reproduction steps only when they are safe to share.
6. Avoid posting secrets, tokens or private data.
7. Avoid publishing weaponized proof-of-concept code.
8. Allow maintainers time to investigate and remediate.
```

Non-sensitive security documentation improvements may be submitted through normal pull requests.

Sensitive vulnerabilities should not be disclosed publicly until reviewed.

```txt

risk = classify sensitivity and potential impact
human oversight = preserve human accountability
decision = ALLOW, BLOCK, ESCALATE, DEGRADE, AUDIT or NOOP
execution = perform only the permitted safe operation
EVT = record relevant operation
memory = preserve EVT/IPR-bound continuity when required
OPC = create proof receipt when required
ledger = preserve continuity
verification = allow reconstruction
continuity = link current state to previous events
```

No sensitive operation should bypass this sequence.

## 13. Fail-Closed Security

AI JOKER-C2 follows a fail-closed security model.

Fail-closed must activate when:

```txt
identity context is missing;
IPR binding is missing where required;
runtime state is invalid;
user authority is unclear;
intent is unsafe;
project collection cannot be classified in a sensitive context;
HBCE module cannot be classified in a sensitive context;
policy cannot be applied;
risk cannot be classified;
requested operation may enable abuse;
requested operation may expose secrets;
requested operation may affect real systems without authorization;
requested operation may affect civic or democratic infrastructure without safeguards;
requested operation may affect high-risk AI governance without oversight;
event trace cannot be generated;
memory continuity cannot be preserved when required;
OPC proof receipt cannot be generated when required;
ledger continuity cannot be preserved;
legal or compliance basis is uncertain;
human review is required but absent;
the system is degraded in a sensitive context.
```

The rule is:

```txt
Uncertainty in sensitive operations does not authorize execution.
```

Fail-closed outcomes:

```txt
BLOCK = unsafe operation is refused
DEGRADE = only safe limited help is provided
ESCALATE = human review or authority is required
AUDIT = operation requires record or review
NOOP = no operational action is taken
```

Fail-closed is controlled safety behavior.

It is not system failure.

## 35. Incident Response

If a security issue is discovered:

```txt
1. Stop unsafe deployment if necessary.
2. Preserve relevant logs and EVT records when safe.
3. Remove exposed secrets from active environments.
4. Rotate compromised credentials.
5. Patch the affected file or route.
6. Add a correction event or changelog note when appropriate.
7. Verify the fix locally.
8. Deploy the fixed version.
9. Review whether documentation or policy must be updated.
10. Avoid public disclosure of exploit details before mitigation.
```

If a secret was committed, removing it from the latest commit is not enough.

The secret should be considered compromised and rotated.
